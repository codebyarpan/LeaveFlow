- (decision) CARRY-FORWARD IS DERIVED, NOT ACCUMULATED (user-chosen). carried_forward(Y+1) = min(carry_forward_cap, Available(Y)), with Available(Y) read live and recomputed whenever year Y's balance changes; propagates forward through all materialized subsequent leave years. Binds: FR-07, DR-7, DR-7a, the rollover job. Prevents: (a) double-counting days that are both Consumed in year Y and carried into Y+1, (b) silently lapsing days released by a post-boundary rejection. Safety proof: approval is a Reserved->Consumed transfer, so Available is invariant under approval; only release raises it. Available(Y) is therefore monotonically non-decreasing and carry-forward can only be topped up, never clawed back -- so no retroactive reduction can violate the CHECK constraint.
- (decision) ROLLOVER IDEMPOTENCE (PRD open question, now closed): falls out of the derived-carry-forward rule. Rollover SETs derived values rather than accumulating, so a second run against the same Leave Year is a no-op by construction. No guard table, no run-once flag.
- (decision) CONCURRENCY MECHANISM (user-chosen, alternatives shown): pessimistic row lock + guarded conditional UPDATE. (1) The service layer opens exactly one transaction per command. (2) It acquires the affected leave_balance row(s) with SELECT ... FOR UPDATE before reading any quantity it will write. (3) A Leave Request / Cancellation Request transition is a conditional UPDATE naming the expected from-state in its WHERE clause; rowcount 0 = refused (this IS FR-09's first-committed-wins). (4) CHECK constraint is the final backstop. Rejected: optimistic version+retry (retry wrapper is where a 3-day project puts its bugs; optimises contention the NFR set declares out of scope); SERIALIZABLE (every command needs a 40001 retry wrapper; every read pays; hardest to defend aloud).
- (decision) LOCK ORDERING: every transaction acquires leave_balance rows before leave_request / cancellation_request rows; where several balance rows are locked (rollover, holiday recalculation), they are locked in ascending (employee_id, leave_type_id, leave_year). Prevents deadlock between concurrent submissions, approvals, and the rollover job.
- (question) PROPAGATION BOUND: nothing in any source document bounds how long a Leave Request may stay Pending. A 2026 request still pending when 2028 is materialized forces carry-forward recomputation to propagate forward across open years. Expiry of long-pending requests is unmodelled.
