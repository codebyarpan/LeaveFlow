## Deferred from: code review of 1-1-project-foundation-and-reproducible-setup (2026-07-10)

- `configure_logging()` runs as an import side effect of `app.main` ([backend/app/main.py:24](../../backend/app/main.py)), replacing the root logger's handlers for any process that imports the module — pytest included (`tests/test_health.py` imports `app.main` at collection). Low consequence today; revisit if `caplog`-based tests arrive or if a host application ever embeds the app. Candidate fix: move configuration behind an app factory or uvicorn `log_config`.
- A top-level pip package literally named `seed` is installed into site-packages (`include = ["app*", "seed*"]` in [backend/pyproject.toml:54](../../backend/pyproject.toml)), and the api image keeps a duplicate source copy at `/srv` on `sys.path` alongside the installed one. `seed` is a collision-prone module name and two copies of a package on `sys.path` is an ambiguity waiting for a symptom — but the story's prescribed source tree fixes `backend/seed/`, so renaming is a spec change, not a patch.

## Deferred from: code review of 1-5-manage-departments (2026-07-11)

- **Enveloped server-side validation for department `name` (and free-text writes generally).** Empty/whitespace names are currently accepted (`POST/PATCH /departments {"name": ""}`), violating the server-as-guard invariant (NFR-03). Blocked on an architecture decision: how validation failures are enveloped in the NFR-17 `{code,message,details}` contract (new vocabulary code + status mapping), since a raw pydantic 422 bypasses it. This is the first free-text write; the convention set here will apply to every later validated field. [backend/app/api/v1/departments.py, backend/app/services/departments.py]
- **Pagination controls for the Departments UI.** The backend pagination convention is fully built (bound 50/100), but the frontend requests only the first page and renders no next/prev controls, so departments past the first page are unmanageable via the UI. Deferred until a real >50-department need appears. [frontend/src/api/departments.ts, frontend/src/features/departments/DepartmentsPage.tsx]

## Deferred from: code review of 1-6-manage-employees-and-reporting-lines (2026-07-11)

- **Pagination controls for the Employees UI, and the Manager `<select>` cap.** `useEmployees()` fetches only the first page (default 50), and both the list and the Manager dropdowns render from that page. With >50 employees, rows 51+ are invisible/unmanageable and a manager beyond row 50 cannot be assigned — a concrete AC13 gap at scale. Same shape as the deferred Departments pagination gap; a shared pagination-UI treatment likely resolves both. [frontend/src/features/employees/EmployeesPage.tsx, frontend/src/api/employees.ts]
- **AD-22 no-orphan invariant is one-directional and non-atomic.** Nothing prevents naming a *deactivated* Employee as a new manager (Open Q3 — allowed by design, flagged as a possible future guard), and the `count_active_direct_reports` deactivation/demotion guard has no DB backstop and no row lock, so a concurrent `POST /employees {manager_id: M}` and `POST /employees/M/deactivate` can both commit and orphan an active report. The orphaning consequence (auto-approval of the report's requests) is vacuous until Epic 2 creates `leave_request`; revisit when that guard becomes executable. [backend/app/services/employee.py]
- **Email uniqueness is case/whitespace-sensitive; no server-side email normalization or format validation.** `get_by_email` is an exact match, `email` is a bare `str` (not `EmailStr`), and the service never trims/lowercases, so `Foo@x.com` and `foo@x.com` are two distinct loginnable accounts and the "one address, one account" intent is not enforced. Pre-existing (the login path shares `get_by_email`); folds into the same enveloped-validation contract deferred for department/employee free-text (Open Q2). [backend/app/services/employee.py, backend/app/repositories/employee.py]
- **A >72-byte initial password 500s.** `create_employee` calls `security.hash_password`, which raises `ValueError` on a password over bcrypt's 72-byte limit; it is not caught, so it surfaces as a 500 rather than an enveloped 4xx. Explicitly sanctioned by Trap 5 / Open Q2 pending the validation-error contract. [backend/app/services/employee.py:162]
- **Unbounded `page` query param.** `PageParams` clamps `page_size` and floors `page` at 1 but sets no ceiling on `page`; a huge `page` makes `offset = (page-1)*page_size` exceed PostgreSQL `bigint` → 500. Pre-existing in the shared pagination module from Story 1.5 (affects `GET /departments` equally). [backend/app/api/v1/pagination.py:82]
